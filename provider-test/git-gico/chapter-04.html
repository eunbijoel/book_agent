4장: 변경 이력 탐색 및 관리

상상해보세요. 여러분이 몇 주 동안 공들여 개발한 프로젝트에서 갑자기 심각한 버그가 발견되었습니다. 문제는 이 버그가 언제, 누구에 의해, 어떤 변경 사항 때문에 발생했는지 알 수 없다는 것입니다. 아니면, 특정 기능이 언제 도입되었고, 그 기능이 어떤 과정을 거쳐 현재의 모습이 되었는지 궁금할 수도 있습니다. 이럴 때, 여러분은 마치 과거로 돌아가 프로젝트의 모든 발자취를 추적할 수 있는 시간 탐정처럼 되고 싶을 것입니다.

Git은 바로 이런 상황을 위해 존재합니다. 지난 장에서 우리는 git add와 git commit을 통해 프로젝트의 특정 순간을 "스냅샷"으로 저장하는 방법을 배웠습니다. 이제 이 스냅샷들이 모여 만들어진 변경 이력을 어떻게 효과적으로 탐색하고, 각 스냅샷 간의 차이점을 비교하며, 프로젝트의 진화를 이해할 수 있는지 알아볼 시간입니다. Git의 강력한 이력 관리 도구들을 통해 여러분은 더 이상 미지의 버그 앞에서 당황하지 않고, 프로젝트의 모든 변화를 명확하게 파악할 수 있게 될 것입니다.

4.1. Git의 시간 여행자: git log

git log 명령어는 Git 저장소의 커밋 이력을 조회하는 가장 기본적인 도구입니다. 이 명령어를 통해 우리는 누가, 언제, 무엇을 변경했는지에 대한 정보를 얻을 수 있습니다.

4.1.1. 기본 git log 사용하기

터미널에서 git log를 입력하면, 가장 최근 커밋부터 역순으로 모든 커밋의 상세 정보가 출력됩니다.

git log

출력되는 정보는 다음과 같습니다:

  • commit 해시 (SHA-1): 각 커밋을 고유하게 식별하는 40자 길이의 영숫자 문자열입니다. 이 해시는 Git에서 커밋을 참조하는 데 사용되는 핵심 식별자입니다. (예: commit 8e5f1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a)
  • Author: 커밋을 생성한 사람의 이름과 이메일 주소입니다.
  • Date: 커밋이 생성된 날짜와 시간입니다.
  • Commit message: 커밋을 생성할 때 작성한 설명 메시지입니다.

예시 출력:

commit 8e5f1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a (HEAD -> main)
Author: Your Name <your.email@example.com>
Date:   Mon Jan 1 10:00:00 2023 +0900

    feat: Add user authentication module

commit 1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b
Author: Another Developer <another.developer@example.com>
Date:   Sun Dec 31 15:30:00 2022 +0900

    docs: Update README with project setup instructions

... (이전 커밋들)

이 기본 출력은 많은 정보를 제공하지만, 프로젝트의 이력이 길어질수록 한눈에 파악하기 어렵다는 단점이 있습니다. 다행히 git log는 다양한 옵션을 제공하여 우리가 원하는 형태로 이력을 조회할 수 있도록 돕습니다.

4.1.2. 간결하게 보기: --oneline

가장 자주 사용되는 옵션 중 하나는 --oneline입니다. 이 옵션은 각 커밋을 한 줄로 요약하여 보여주므로, 전체 이력을 훨씬 간결하게 파악할 수 있습니다.

git log --oneline

예시 출력:

8e5f1b2 (HEAD -> main) feat: Add user authentication module
1a2b3c4 docs: Update README with project setup instructions
f9e8d7c refactor: Improve data validation logic
...

여기서 8e5f1b2는 40자 커밋 해시의 앞 7자리로, Git에서 커밋을 참조할 때 이 축약된 해시를 사용하는 것이 일반적입니다.

4.1.3. 그래프로 보기: --graph

브랜치(branch)와 병합(merge) 이력이 복잡한 프로젝트에서는 커밋들이 시간 순서대로만 나열되는 것만으로는 충분하지 않습니다. --graph 옵션은 커밋 이력을 ASCII 아트 그래프로 시각화하여 브랜치의 분기 및 병합 과정을 한눈에 보여줍니다.

git log --oneline --graph

이 옵션은 보통 --oneline과 함께 사용되며, 모든 브랜치의 이력을 보려면 --all 옵션을 추가할 수 있습니다.

git log --oneline --graph --all

예시 출력:

* 8e5f1b2 (HEAD -> main) feat: Add user authentication module
*   c7d6e5f Merge branch 'feature/payment'
|\
| * 4b3a210 feat: Implement payment gateway
* | 1a2b3c4 docs: Update README with project setup instructions
|/
* f9e8d7c refactor: Improve data validation logic
...

이 그래프를 통해 feature/payment 브랜치가 main 브랜치에서 분기되었고, 나중에 main으로 병합되는 과정을 시각적으로 이해할 수 있습니다.

4.1.4. 사용자 정의 형식: --pretty=format

--pretty=format:<string> 옵션을 사용하면 git log의 출력 형식을 완전히 사용자 정의할 수 있습니다. 이는 특정 정보만 추출하거나, 보고서 형식으로 이력을 출력할 때 매우 유용합니다.

자주 사용되는 형식 지정자(placeholder)는 다음과 같습니다:

  • %h: 축약된 커밋 해시
  • %H: 전체 커밋 해시
  • %an: 저자 이름
  • %ar: 저자 날짜 (상대적 형식, 예: "2 weeks ago")
  • %ad: 저자 날짜 (기본 형식)
  • %s: 커밋 메시지 제목
  • %b: 커밋 메시지 본문

더 많은 형식 지정자는 Git 공식 문서(git help log 참조)에서 확인할 수 있습니다.

예시: 커밋 해시, 저자 이름, 상대적 날짜, 커밋 메시지를 한 줄로 출력하고 싶을 때:

git log --pretty=format:"%h - %an, %ar : %s"

예시 출력:

8e5f1b2 - Your Name, 2 minutes ago : feat: Add user authentication module
1a2b3c4 - Another Developer, 1 hour ago : docs: Update README with project setup instructions
f9e8d7c - Your Name, 3 hours ago : refactor: Improve data validation logic

4.1.5. 이력 필터링하기

git log는 단순히 모든 이력을 보여주는 것을 넘어, 우리가 관심 있는 커밋만 필터링하여 보여주는 강력한 기능을 제공합니다.

  • 특정 저자의 커밋 조회: --author 옵션을 사용하여 특정 저자가 작성한 커밋만 볼 수 있습니다. bash git log --author="Your Name"
  • 특정 문자열을 포함하는 커밋 메시지 조회: --grep 옵션은 커밋 메시지에서 특정 패턴을 검색합니다. bash git log --grep="feature"
  • 특정 파일의 변경 이력 조회: 특정 파일의 변경 이력만 보고 싶을 때 유용합니다. git log 명령어 뒤에 -- 구분자와 파일 경로를 지정합니다. bash git log -- path/to/your/file.txt 예를 들어, src/index.js 파일의 이력만 보고 싶다면: bash git log -- src/index.js
  • 커밋 개수 제한: -n <count> 옵션을 사용하여 최신 count개의 커밋만 조회할 수 있습니다. bash git log -n 3 # 최신 3개의 커밋 조회
  • 날짜 범위 지정: --since (이후)와 --until (이전) 옵션을 사용하여 특정 기간 내의 커밋만 조회할 수 있습니다. bash git log --since="2023-01-01" --until="2023-01-31" # 2023년 1월 한 달간의 커밋 git log --since="2 weeks ago" # 2주 전부터 현재까지의 커밋

실습: git log 마스터하기

  1. 새로운 Git 저장소를 초기화하거나, 기존 저장소에서 작업합니다.
  2. index.html 파일을 생성하고 내용을 추가한 후 첫 커밋을 합니다.
  3. style.css 파일을 생성하고 내용을 추가한 후 두 번째 커밋을 합니다.
  4. index.html 파일을 수정하고 세 번째 커밋을 합니다.
  5. git log의 다양한 옵션(--oneline, --graph --all, --pretty=format:"%h %s", -- index.html)을 사용하여 커밋 이력을 조회해보세요.
  6. git log --author="Your Name"과 같이 여러분의 이름으로 필터링하여 커밋을 찾아보세요.

4.2. 변경 사항 추적하기: git diff

git log가 "무엇이 변경되었는지"에 대한 요약 정보를 제공한다면, git diff는 "정확히 어떤 내용이 변경되었는지"를 보여주는 명령어입니다. git diff는 작업 디렉토리, 스테이징 영역, 그리고 커밋 간의 차이점을 비교하는 데 사용됩니다.

오해 풀기: 많은 사람들이 Git이 파일의 변경 사항(diff)만 저장한다고 생각하지만, 실제로는 각 커밋마다 프로젝트의 전체 스냅샷을 저장합니다. git diff는 이 스냅샷들을 비교하여 변경 사항을 계산하고 보여주는 도구입니다.

4.2.1. 작업 디렉토리 vs. 스테이징 영역

이것은 가장 기본적인 git diff 사용법입니다. 아직 git add 하지 않은, 즉 작업 디렉토리에서 수정한 내용과 스테이징 영역에 있는 내용(마지막 git add 시점)을 비교합니다.

git diff

이 명령은 여러분이 파일을 수정한 후 아직 git add 하지 않은 변경 사항들을 보여줍니다.

예시 시나리오: 1. index.html 파일을 수정합니다. 2. git status를 입력하면 "Changes not staged for commit" 섹션에 index.html이 나타날 것입니다. 3. git diff를 입력하면 index.html 파일의 수정 내용이 출력됩니다.

4.2.2. 스테이징 영역 vs. 마지막 커밋

git diff --staged 또는 git diff --cached 명령어는 스테이징 영역에 있는 변경 사항(즉, git add로 추가했지만 아직 커밋하지 않은 내용)과 가장 최근 커밋 간의 차이점을 비교합니다.

git diff --staged
# 또는
git diff --cached

이 명령은 여러분이 커밋할 준비가 된 변경 사항들을 미리 검토할 때 유용합니다.

예시 시나리오: 1. index.html 파일을 수정합니다. 2. git add index.html을 통해 변경 사항을 스테이징 영역에 추가합니다. 3. git status를 입력하면 "Changes to be committed" 섹션에 index.html이 나타날 것입니다. 4. git diff --staged를 입력하면 스테이징 영역에 있는 index.html의 변경 내용이 출력됩니다.

4.2.3. 두 커밋 간의 변경 사항

git diff <commit1> <commit2> 명령어를 사용하여 두 커밋 간의 모든 변경 사항을 비교할 수 있습니다. 여기서 <commit1>과 <commit2>는 커밋 해시(축약된 해시도 가능), 브랜치 이름, 태그 이름 또는 HEAD와 같은 참조(reference)가 될 수 있습니다.

  • 마지막 커밋과 그 이전 커밋 비교: bash git diff HEAD~1 HEAD HEAD는 현재 작업 중인 브랜치의 최신 커밋을 가리키는 포인터이고, HEAD~1은 HEAD의 바로 이전 커밋을 의미합니다.
  • 특정 두 커밋 간 비교: bash git diff 1a2b3c4 8e5f1b2 (1a2b3c4와 8e5f1b2는 실제 커밋 해시로 대체해야 합니다.)
  • 특정 파일의 두 커밋 간 변경 사항 비교: bash git diff <commit_hash_1> <commit_hash_2> -- <file_path> 예를 들어: bash git diff 1a2b3c4 8e5f1b2 -- src/index.js

4.2.4. git diff 출력 해석하기

git diff의 출력은 일반적으로 다음과 같은 형식으로 나타납니다.

diff --git a/index.html b/index.html
index 1234567..abcdefg 100644
--- a/index.html
+++ b/index.html
@@ -1,3 +1,4 @@
 <!DOCTYPE html>
 <html lang="ko">
 <head>
-    <meta charset="UTF-8">
+    <meta charset="UTF-8"> <!-- 주석 추가 -->
     <title>My Web Page</title>
</head>
  • diff --git a/index.html b/index.html: a/index.html과 b/index.html 파일을 비교하고 있다는 것을 나타냅니다. a는 이전 버전, b는 새 버전을 의미합니다.
  • index 1234567..abcdefg 100644: Git 내부적인 정보로, 파일의 해시 값과 파일 모드를 나타냅니다.
  • --- a/index.html: 이전 버전 파일의 경로.
  • +++ b/index.html: 새 버전 파일의 경로.
  • @@ -1,3 +1,4 @@: "청크 헤더(chunk header)"라고 불립니다.
    • -1,3: 이전 파일에서 1번째 줄부터 3줄이 변경되었음을 의미합니다.
    • +1,4: 새 파일에서 1번째 줄부터 4줄이 변경되었음을 의미합니다.
  • +로 시작하는 줄: 새 버전에서 추가된 내용.
  • -로 시작하는 줄: 이전 버전에서 삭제된 내용.
  • 공백으로 시작하는 줄: 변경되지 않은 내용 (컨텍스트를 제공).

실습: git diff 활용하기

  1. 이전 실습에서 사용했던 Git 저장소로 이동합니다.
  2. index.html 파일의 내용을 일부 수정합니다.
  3. git diff 명령어를 사용하여 아직 스테이징하지 않은 변경 사항을 확인합니다.
  4. git add index.html을 실행하여 변경 사항을 스테이징 영역에 추가합니다.
  5. git diff 명령어를 다시 실행하여 아무것도 출력되지 않음을 확인합니다 (작업 디렉토리와 스테이징 영역이 동일하므로).
  6. git diff --staged 명령어를 사용하여 스테이징 영역에 있는 변경 사항을 확인합니다.
  7. 새로운 커밋을 생성합니다.
  8. git diff HEAD~1 HEAD 명령어를 사용하여 방금 생성한 커밋과 그 이전 커밋 간의 차이점을 확인합니다.

4.3. Git의 내부 작동 방식: 커밋 해시와 HEAD

git log와 git diff를 효과적으로 사용하려면 Git이 변경 이력을 내부적으로 어떻게 관리하는지에 대한 기본적인 이해가 필요합니다. 핵심 개념은 커밋 해시와 HEAD 포인터입니다.

4.3.1. 커밋 해시

모든 커밋은 40자 길이의 고유한 해시(현재 SHA-1 알고리즘 기반)를 가집니다. 이 해시는 커밋의 내용(파일 스냅샷), 메타데이터(저자, 날짜, 메시지), 그리고 부모 커밋의 해시를 기반으로 계산됩니다.

  • 고유성: 단 하나의 커밋만이 특정 해시를 가질 수 있습니다. 이는 커밋의 무결성을 보장합니다.
  • 불변성: 커밋 내용이 변경되면 해시도 변경됩니다. 따라서 한 번 커밋된 내용은 변경될 수 없으며, 만약 변경하려 한다면 새로운 해시를 가진 새로운 커밋이 생성됩니다.
  • 참조: 우리는 이 해시를 사용하여 특정 커밋을 정확하게 지칭하고, 그 커밋으로 이동하거나 다른 커밋과 비교할 수 있습니다. git log에서 본 축약된 7자리 해시도 충분히 고유하여 보통 사용됩니다.

4.3.2. HEAD 포인터

HEAD는 Git에서 매우 중요한 개념입니다. 이는 현재 작업 중인 브랜치 또는 현재 체크아웃된 커밋을 가리키는 포인터입니다.

  • 대부분의 경우, HEAD는 현재 브랜치를 가리킵니다. 예를 들어, main 브랜치에서 작업 중이라면 HEAD는 main 브랜치를 가리키고, main 브랜치는 다시 최신 커밋을 가리킵니다.
  • 새로운 커밋을 생성하면, HEAD가 가리키는 브랜치와 해당 브랜치가 가리키는 커밋이 함께 앞으로 이동합니다.
  • git checkout <branch_name> 명령으로 브랜치를 전환하면, HEAD는 새로운 브랜치를 가리키게 됩니다.
  • git checkout <commit_hash> 명령으로 특정 커밋을 직접 체크아웃하면, HEAD는 더 이상 브랜치를 가리키지 않고 특정 커밋을 직접 가리키게 됩니다. 이 상태를 '분리된 HEAD(detached HEAD)' 상태라고 합니다. 이 상태에서 커밋을 하면, 해당 커밋은 어떤 브랜치에도 속하지 않게 됩니다.

HEAD의 이동 시각화:

[main]
  |
  V
(C1) -- (C2) -- (C3) -- (C4)  <- HEAD (main 브랜치에서 작업 중)

만약 feature 브랜치를 만들고 전환하면:

[main]
  |
  V
(C1) -- (C2) -- (C3) -- (C4)
                      \
                       (C5)  <- HEAD (feature 브랜치에서 작업 중)
                         ^
                       [feature]

4.3.3. Git 객체 모델: 스냅샷의 마법 (심화 개념)

Git은 변경 이력을 저장할 때, 파일의 변경 사항(diff)을 저장하는 것이 아니라, 각 커밋마다 프로젝트의 전체 스냅샷을 저장합니다. 물론, 효율적인 저장 공간 관리를 위해 파일이 변경되지 않은 경우 이전 스냅샷의 파일을 재사용하는 영리한 방식으로 작동합니다.

Git은 이 스냅샷을 저장하기 위해 세 가지 핵심 객체 유형을 사용합니다.

  1. Blob (블롭) 객체:

    • 파일의 실제 내용(데이터)을 저장합니다.
    • 파일 이름이나 권한 정보는 포함되지 않습니다.
    • 동일한 내용의 파일이 여러 개 있어도 Git은 Blob 객체를 하나만 저장하고 여러 곳에서 참조하여 저장 공간을 절약합니다.

  2. Tree (트리) 객체:

    • 디렉토리 구조와 파일 이름을 저장합니다.
    • Blob 객체(파일) 또는 다른 Tree 객체(하위 디렉토리)에 대한 참조를 포함합니다.
    • 각 Tree 객체는 특정 시점의 디렉토리 구조 스냅샷을 나타냅니다.

  3. Commit (커밋) 객체:

    • 특정 시점의 프로젝트 스냅샷입니다.
    • 다음과 같은 정보를 포함합니다:
      • 현재 프로젝트의 루트 디렉토리를 나타내는 Tree 객체에 대한 참조.
      • 이전 커밋(부모 커밋)의 커밋 해시. (최초 커밋은 부모가 없습니다. 병합 커밋은 여러 부모를 가질 수 있습니다.)
      • 커밋을 만든 저자 정보.
      • 커밋이 생성된 날짜와 시간.
      • 커밋에 대한 메시지.

이러한 객체 모델 덕분에 Git은 매우 강력한 버전 관리 시스템이 될 수 있습니다. 각 커밋이 완전한 스냅샷을 나타내므로, 언제든지 특정 커밋으로 돌아가 프로젝트의 상태를 정확히 재현할 수 있습니다.

반성 및 토론:

  • git log의 어떤 필터링 옵션이 버그를 추적하거나 특정 기능의 도입 시점을 파악하는 데 가장 유용할 것이라고 생각하나요?
  • git diff의 세 가지 주요 사용 사례(작업 디렉토리 vs. 스테이징, 스테이징 vs. 커밋, 커밋 vs. 커밋) 중 어떤 것이 일상적인 개발 흐름에서 가장 자주 사용될까요? 각 상황에서 왜 그 명령어가 필요한지 설명해보세요.
  • HEAD가 main 브랜치를 가리키는 것과, 특정 커밋 해시를 직접 가리키는 것(detached HEAD)의 차이점은 무엇일까요?

요약 및 다음 장 미리보기

이번 장에서는 Git 저장소의 변경 이력을 탐색하고 이해하는 데 필수적인 도구들을 살펴보았습니다.

  • git log를 통해 커밋 해시, 저자, 날짜, 메시지 등 커밋의 상세 정보를 조회하고, --oneline, --graph, --pretty=format과 같은 다양한 옵션으로 출력을 간결하게 만들거나 시각화하는 방법을 배웠습니다. 또한, --author, --grep, -- <file_path> 옵션으로 이력을 필터링하는 방법도 익혔습니다.
  • git diff를 사용하여 작업 디렉토리, 스테이징 영역, 그리고 커밋 간의 변경 사항을 비교하는 방법을 알아보았고, 그 출력 결과를 해석하는 법도 이해했습니다.
  • Git의 내부 작동 방식에 대한 이해를 돕기 위해, 각 커밋을 고유하게 식별하는 커밋 해시(SHA-1 기반)와 현재 작업 위치를 가리키는 HEAD 포인터의 역할을 학습했습니다. 나아가, Git이 Blob, Tree, Commit 객체를 사용하여 프로젝트의 스냅샷을 효율적으로 저장하는 기본적인 원리도 간략하게 다루었습니다.

이제 여러분은 Git 저장소의 역사를 읽고, 특정 변경 사항을 찾아내며, 프로젝트의 진화를 이해할 수 있는 강력한 능력을 갖추게 되었습니다. 하지만 때로는 과거를 읽는 것을 넘어, 과거를 '수정'하거나 '되돌리고' 싶을 때가 있습니다. 다음 장에서는 실수로 잘못 커밋했거나, 특정 변경 사항을 취소해야 할 때 Git이 제공하는 안전하고 강력한 "변경 사항 되돌리기" 도구들에 대해 자세히 알아보겠습니다. 과거의 실수를 바로잡고, 깨끗한 이력을 유지하는 방법을 함께 배워봅시다!
