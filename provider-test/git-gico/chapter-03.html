3장: 로컬 저장소 시작하기: 커밋의 기본

상상해 보세요. 야심 차게 시작한 새로운 프로젝트의 첫 번째 코드를 작성하고, 몇 시간 동안 수정과 개선을 거듭한 끝에 마침내 만족스러운 결과물을 얻었습니다. 이제 이 소중한 작업물을 안전하게 저장해야 할 때입니다. 그런데 단순히 파일을 저장하는 것만으로는 부족합니다. 만약 나중에 이 코드를 다시 수정해야 하는데, 이전 버전으로 되돌리고 싶다면 어떻게 해야 할까요? 실수로 중요한 부분을 삭제하거나, 팀원과 함께 작업하다가 변경 이력이 꼬인다면요?

이러한 문제들을 해결하고 여러분의 작업물을 안전하게 지켜줄 Git의 가장 기본적인 마법, 바로 '커밋(Commit)'을 배울 시간입니다. 커밋은 여러분의 프로젝트가 특정 시점에 어떤 모습이었는지를 기록하는 스냅샷과 같습니다. 마치 사진작가가 중요한 순간을 포착하듯, Git은 여러분의 코드 변경 이력을 정확히 포착하고 영구적으로 저장합니다. 이 장에서는 새로운 Git 저장소를 만들고, 파일 변경 사항을 추적하며, 의미 있는 커밋을 통해 변경 이력을 차곡차곡 쌓아가는 방법을 자세히 살펴보겠습니다.

3.1 새로운 Git 저장소 초기화: git init

Git을 사용하기 위한 첫 번째 단계는 프로젝트 폴더를 Git이 관리하는 '저장소(Repository)'로 만드는 것입니다. 저장소는 Git이 프로젝트의 모든 변경 이력을 추적하고 저장하는 특별한 공간입니다. 이 과정을 '초기화(Initialize)'라고 부르며, git init 명령어를 통해 수행합니다.

git init은 여러분의 현재 작업 디렉토리(Working Directory)를 Git 저장소로 변환합니다. 이 명령어를 실행하면, Git은 해당 디렉토리 안에 .git이라는 숨겨진 하위 디렉토리를 생성합니다. 이 .git 디렉토리는 여러분의 프로젝트에 대한 모든 Git 관련 정보, 즉 변경 이력, 설정, 객체 데이터베이스 등을 포함하고 있습니다. .git 디렉토리는 Git 저장소의 핵심이며, 절대로 수동으로 건드리거나 삭제해서는 안 됩니다.

그럼 직접 해볼까요?

  1. 새로운 프로젝트 폴더 생성: 터미널 또는 명령 프롬프트를 열고, Git 저장소로 만들고 싶은 폴더를 생성합니다. bash mkdir my_new_project cd my_new_project 이 명령은 my_new_project라는 새 폴더를 만들고, 그 안으로 이동합니다.

  2. Git 저장소 초기화: 이제 이 폴더 안에서 git init 명령어를 실행합니다. bash git init 명령어를 실행하면 다음과 유사한 메시지가 나타날 것입니다: Initialized empty Git repository in /path/to/my_new_project/.git/ 이 메시지는 현재 디렉토리가 성공적으로 Git 저장소로 초기화되었음을 의미합니다.

이제 my_new_project 폴더는 단순한 폴더가 아니라, Git의 관리를 받는 '로컬 저장소(Local Repository)'가 되었습니다. 앞으로 이 안에서 이루어지는 모든 파일 변경 사항은 Git에 의해 추적될 준비가 된 것입니다.

3.2 Git의 세 가지 상태와 작업 영역

Git의 핵심적인 작동 방식을 이해하려면, 파일이 Git의 관리 하에 놓일 때 거치는 세 가지 주요 상태와 그에 해당하는 세 가지 작업 영역을 알아야 합니다. 이 개념은 Git의 버전 관리 흐름을 파악하는 데 매우 중요합니다.

사진작가의 작업 과정을 예로 들어볼까요? * 작업 디렉토리 (Working Directory): 카메라를 들고 현장에서 사진을 찍는 순간입니다. 아직 어떤 사진을 갤러리에 올릴지 결정되지 않은, 현재 작업 중인 상태입니다. * 스테이징 영역 (Staging Area / Index): 찍은 사진들 중 "이건 정말 잘 나왔다! 갤러리에 올릴 후보군"이라고 선별하여 임시로 모아두는 곳입니다. 아직 최종 갤러리에 등록된 것은 아니지만, 다음 번 등록에 포함될 사진들입니다. * 로컬 저장소 (Local Repository): 선별된 사진들을 최종적으로 갤러리에 등록하여 영구적으로 보관하는 곳입니다. 한 번 등록된 사진은 언제든 다시 찾아볼 수 있습니다.

Git에서의 세 가지 상태와 작업 영역은 다음과 같습니다:

  1. 작업 디렉토리 (Working Directory):

    • 파일 상태: 이 영역의 파일은 주로 수정됨 (Modified) 또는 추적되지 않음 (Untracked) 상태입니다.
    • 여러분 눈에 보이는 프로젝트 파일들이 위치한 곳입니다.
    • 파일을 생성하거나, 수정하거나, 삭제하는 모든 작업이 여기서 이루어집니다.
    • Git이 아직 추적하지 않는 새 파일은 '추적되지 않음(Untracked)' 상태이며, Git이 추적하던 파일이 변경되면 '수정됨(Modified)' 상태가 됩니다.

  2. 스테이징 영역 (Staging Area / Index):

    • 파일 상태: 이 영역에 추가된 파일은 스테이지됨 (Staged) 상태입니다.
    • 다음 커밋에 포함하고 싶은 변경 사항들을 임시로 모아두는 곳입니다.
    • Git은 이 영역에 있는 파일들만 커밋 대상으로 간주합니다.
    • 이 영역은 유연성을 제공합니다. 여러 파일 중 일부만 커밋하거나, 한 파일의 변경 사항 중 일부만 커밋하는 것도 가능하게 합니다.

  3. 로컬 저장소 (Local Repository):

    • 파일 상태: 이 영역에 저장된 파일은 커밋됨 (Committed) 상태입니다.
    • 스테이징 영역에 있는 파일들의 스냅샷이 영구적으로 저장되는 곳입니다.
    • 각 커밋은 고유한 식별자(해시)를 가지며, 특정 시점의 프로젝트 전체 상태를 나타냅니다.
    • 한 번 커밋된 내용은 언제든지 되돌아볼 수 있습니다.

이 세 가지 영역을 이해하는 것이 Git을 효과적으로 사용하는 데 필수적입니다. 여러분은 작업 디렉토리에서 코드를 개발하고, git add 명령어를 통해 변경 사항을 스테이징 영역으로 옮긴 다음, git commit 명령어로 스테이징 영역의 내용을 로컬 저장소에 영구적으로 기록하게 됩니다.

3.3 파일 변경 추적 및 스테이징: git status와 git add

이제 실제로 파일을 만들고 Git이 어떻게 변경 사항을 추적하는지 살펴보겠습니다.

3.3.1 현재 상태 확인하기: git status

git status 명령어는 현재 작업 디렉토리와 스테이징 영역의 상태를 요약해서 보여주는 매우 유용한 도구입니다. Git 작업을 할 때 거의 매 순간 사용하게 될 것입니다.

방금 초기화한 my_new_project 폴더 안에서 git status를 실행해 보세요.

git status

아마 다음과 같은 메시지가 나타날 것입니다:

On branch master

No commits yet

nothing to commit (create/copy files and use "git add" to track)
  • On branch master: 현재 master 브랜치에 있다는 의미입니다. 브랜치에 대해서는 나중에 자세히 다룰 것입니다.
  • No commits yet: 아직 어떤 커밋도 없다는 것을 알려줍니다.
  • nothing to commit: 스테이징 영역에 커밋할 내용이 없다는 뜻입니다.

3.3.2 새 파일 추가 및 스테이징: git add

이제 my_new_project 폴더 안에 README.md 파일을 생성하고 내용을 추가해 봅시다.

  1. 파일 생성 및 내용 추가: bash touch README.md echo "# My Awesome Project" > README.md echo "This project is about learning Git basics." >> README.md 이 명령은 README.md 파일을 만들고, 두 줄의 내용을 추가합니다.

  2. 상태 확인: 파일을 생성했으니 다시 git status를 실행해 보세요. bash git status 이번에는 다른 메시지가 나타날 것입니다: ``` On branch master

    No commits yet

    Untracked files: (use "git add ..." to include in what will be committed)

    README.md

    nothing added to commit but untracked files present (use "git add" to track) `` Git은README.md` 파일을 '추적되지 않는 파일(Untracked files)'로 인식했습니다. 이는 Git이 이 파일의 존재를 알고 있지만, 아직 버전 관리 대상으로는 포함하지 않았다는 의미입니다. Git에게 이 파일을 추적하라고 알려줘야 합니다.

  3. 파일 스테이징: git add 명령어를 사용하여 README.md 파일을 스테이징 영역에 추가합니다. bash git add README.md

  4. 다시 상태 확인: git add 후에는 항상 git status로 변경된 상태를 확인하는 습관을 들이는 것이 좋습니다. bash git status ``` On branch master

    No commits yet

    Changes to be committed: (use "git rm --cached ..." to unstage)

    new file:   README.md

    `` 이제README.md파일은 '커밋될 변경 사항(Changes to be committed)' 목록에 나타납니다. 이는README.md` 파일이 작업 디렉토리에서 스테이징 영역으로 옮겨졌으며, 다음 커밋에 포함될 준비가 되었음을 의미합니다.

    오해 해소: 많은 초보자들이 git add를 마치 "파일 저장" 버튼처럼 생각합니다. 하지만 git add는 파일을 로컬 저장소에 직접 저장하는 것이 아니라, 다음 커밋에 포함될 변경 사항들을 '스테이징 영역'에 준비시키는 역할을 합니다.

3.3.3 파일 수정 및 재스테이징

이제 README.md 파일을 수정해보고, Git이 어떻게 이를 처리하는지 확인해 봅시다.

  1. 파일 수정: README.md 파일에 한 줄을 더 추가합니다. bash echo "This is the first commit." >> README.md

  2. 상태 확인: bash git status ``` On branch master

    No commits yet

    Changes to be committed: (use "git rm --cached ..." to unstage)

    new file:   README.md

    Changes not staged for commit: (use "git add ..." to update what will be committed) (use "git restore ..." to discard changes in working directory)

    modified:   README.md

    `` 흥미롭죠? Git은 두 가지 상태를 보여줍니다: *Changes to be committed: 스테이징 영역에 있는README.md파일의 이전 버전 (두 줄이 포함된 버전) *Changes not staged for commit: 작업 디렉토리에 있는README.md` 파일의 현재 버전 (세 줄이 포함된, 방금 수정한 버전)

    이것은 Git의 강력함 중 하나입니다. 여러분은 파일의 수정 사항 중 일부만 스테이징하고 커밋할 수 있습니다. 예를 들어, 여러 기능을 개발하다가 한 기능만 먼저 커밋하고 싶을 때 유용합니다.

  3. 수정 사항 스테이징: 방금 추가한 세 번째 줄까지 커밋에 포함하고 싶다면, 다시 git add README.md를 실행해야 합니다. bash git add README.md 또는 현재 디렉토리의 모든 변경 사항(새 파일, 수정된 파일, 삭제된 파일)을 한 번에 스테이징 영역에 추가하려면 git add . (점 하나) 명령어를 사용합니다. bash git add .

  4. 최종 상태 확인: bash git status ``` On branch master

    No commits yet

    Changes to be committed: (use "git rm --cached ..." to unstage)

    new file:   README.md

    `` 이제README.md` 파일의 최신 수정 사항이 스테이징 영역에 반영되었으며, 다음 커밋에 포함될 준비가 완료되었습니다.

3.4 변경 사항 저장: git commit

스테이징 영역에 변경 사항들이 준비되었다면, 이제 git commit 명령어를 사용하여 이 스냅샷을 로컬 저장소에 영구적으로 기록할 차례입니다.

3.4.1 첫 번째 커밋하기

스테이징 영역에 있는 README.md 파일을 커밋해 봅시다. 커밋할 때는 항상 '커밋 메시지(Commit Message)'를 함께 작성해야 합니다. 커밋 메시지는 이 커밋이 어떤 변경 사항을 포함하고 있는지 설명하는 짧고 명확한 요약입니다.

가장 간단한 커밋 명령어는 -m 옵션을 사용하여 메시지를 직접 지정하는 것입니다.

git commit -m "Initial commit: Add README.md with project description"

명령어를 실행하면 다음과 유사한 메시지가 나타날 것입니다:

[master (root-commit) 6e2f1a3] Initial commit: Add README.md with project description
 1 file changed, 3 insertions(+)
 create mode 100644 README.md
  • [master (root-commit) 6e2f1a3]: master 브랜치에 첫 번째 커밋이 생성되었으며, 6e2f1a3은 이 커밋의 고유한 식별자(커밋 해시의 일부)입니다.
  • 1 file changed, 3 insertions(+): 하나의 파일이 변경되었고, 세 줄이 추가되었음을 나타냅니다.
  • create mode 100644 README.md: README.md 파일이 새로 생성되었음을 보여줍니다.

이제 git status를 다시 실행해 보세요.

git status

On branch master
nothing to commit, working tree clean

nothing to commit, working tree clean 메시지는 현재 작업 디렉토리와 스테이징 영역이 로컬 저장소의 최신 커밋과 동일한 상태이며, 더 이상 커밋할 변경 사항이 없음을 의미합니다. 여러분은 성공적으로 첫 번째 커밋을 만들었습니다!

오해 해소: git commit은 작업 디렉토리의 모든 변경 사항을 저장하는 것이 아니라, 스테이징 영역에 있는 내용만 저장합니다. 만약 git add를 하지 않고 git commit을 실행하면, 스테이징 영역이 비어있으므로 커밋할 내용이 없다는 메시지가 나올 것입니다.

3.4.2 의미 있는 커밋 메시지 작성법

커밋 메시지는 Git 사용의 핵심이자, 팀 협업의 중요한 요소입니다. 좋은 커밋 메시지는 나중에 변경 이력을 추적하거나 문제를 해결할 때 시간을 절약해 줍니다.

좋은 커밋 메시지의 원칙:

  1. 제목 줄 (Subject Line):

    • 간결하게 요약: 50자 이내로 작성합니다.
    • 명령형 동사 사용: "Add feature", "Fix bug", "Update documentation"과 같이 변경 사항의 의도를 명확히 전달하는 명령형 동사로 시작합니다. (예: "기능 추가", "버그 수정", "문서 업데이트")
    • 대문자로 시작: 일반적으로 제목의 첫 글자는 대문자로 시작합니다.
    • 마침표 사용 금지: 제목 줄 끝에는 마침표를 찍지 않습니다.

  2. 본문 (Body):

    • 한 줄 띄기: 제목 줄과 본문 사이에는 반드시 한 줄을 비워둡니다.
    • 상세 설명: 변경 사항의 배경, 이유, 무엇을 해결했는지, 어떤 영향을 미치는지 등을 자세히 설명합니다. 본문은 명령형이 아닌 서술형으로 작성하는 것이 일반적입니다.
    • 72자 이내 줄 바꿈: 가독성을 위해 한 줄에 72자를 넘지 않도록 합니다.

예시:

  • 좋은 커밋 메시지: ``` Feat: Add user authentication module

    This commit introduces a new user authentication module. It includes user registration, login, and session management. Uses JWT for token-based authentication. Resolves issue #123. (한글 예시) Feat: 사용자 인증 모듈 추가

    새로운 사용자 인증 모듈을 추가합니다. 회원가입, 로그인, 세션 관리를 포함합니다. 토큰 기반 인증을 위해 JWT를 사용합니다. 이슈 #123 해결. ```

  • 나쁜 커밋 메시지: 수정함 (무엇을 수정했는지 알 수 없음)

    README.md 파일 업데이트 및 설정 변경 (너무 일반적이고, 왜 변경했는지 알 수 없음)

    버그 수정 완료. 이제 잘 될 거예요. (비격식적이고, 어떤 버그인지 불분명)

git commit 명령어를 -m 옵션 없이 실행하면, 설정된 기본 텍스트 편집기(vi, nano 등)가 열리며 커밋 메시지를 작성할 수 있습니다. 이때 위 원칙에 따라 제목과 본문을 작성하고 저장하면 됩니다.

3.4.3 두 번째 커밋 만들기

README.md 파일을 다시 수정하고 새로운 커밋을 만들어 봅시다.

  1. 파일 수정: bash echo "" >> README.md # 빈 줄 추가 echo "## Getting Started" >> README.md echo "Clone the repository and run `npm install`." >> README.md

  2. 상태 확인: bash git status ``` On branch master Changes not staged for commit: (use "git add ..." to update what will be committed) (use "git restore ..." to discard changes in working directory)

    modified:   README.md

    no changes added to commit (use "git add" and/or "git commit -a)") ``README.md` 파일이 '수정됨(modified)' 상태로, 아직 스테이징 영역에 추가되지 않았음을 보여줍니다.

  3. 스테이징 및 커밋: bash git add README.md git commit -m "Docs: Add getting started section to README" [master 4f6d3b2] Docs: Add getting started section to README 1 file changed, 3 insertions(+) 두 번째 커밋이 성공적으로 생성되었습니다. 이제 git status를 실행하면 다시 working tree clean 상태가 될 것입니다.

3.4.4 실습: 여러 파일 중 일부만 커밋하기

이해를 돕기 위해, 여러 파일이 변경되었을 때 특정 파일만 커밋하는 시나리오를 연습해 봅시다.

  1. 두 개의 새 파일 생성: ```bash touch index.html echo "

    Hello Git!

    " > index.html

    touch style.css echo "body { font-family: sans-serif; }" > style.css ```

  2. 상태 확인: bash git status ``` On branch master Untracked files: (use "git add ..." to include in what will be committed)

    index.html
style.css

    nothing added to commit but untracked files present ``` 두 파일 모두 '추적되지 않음' 상태입니다.

  3. index.html만 스테이징하고 커밋: bash git add index.html git commit -m "Feat: Add basic index.html" [master 9a1b2c3] Feat: Add basic index.html 1 file changed, 1 insertion(+) create mode 100644 index.html

  4. 상태 확인: bash git status ``` On branch master Untracked files: (use "git add ..." to include in what will be committed)

    style.css

    nothing added to commit but untracked files present ``index.html은 커밋되었지만,style.css는 여전히 '추적되지 않음' 상태로 남아 있습니다.git commit은 오직 스테이징 영역에 있는index.html`만 저장소에 기록했음을 알 수 있습니다.

  5. style.css 스테이징 및 커밋: bash git add style.css git commit -m "Style: Add basic styling for body" [master 5d4e3f2] Style: Add basic styling for body 1 file changed, 1 insertion(+) create mode 100644 style.css 이제 모든 파일이 커밋되었고, 작업 디렉토리는 깨끗해졌습니다. 이처럼 git add와 git commit을 통해 원하는 변경 사항만 묶어서 커밋할 수 있습니다.

3.5 Git의 핵심 워크플로우: 세 가지 영역의 상호작용

앞서 살펴본 Git의 세 가지 핵심 영역은 서로 유기적으로 연결되어 프로젝트의 변경 이력을 관리하는 일련의 워크플로우를 형성합니다. 이들의 상호작용을 이해하는 것이 Git을 효과적으로 활용하는 데 필수적입니다.

여러분은 먼저 작업 디렉토리에서 코드를 작성하고 수정합니다. 여기서 파일들은 '수정됨' 또는 '추적되지 않음' 상태가 됩니다. 다음으로, git add 명령어를 사용하여 다음 커밋에 포함하고 싶은 변경 사항들을 스테이징 영역으로 옮깁니다. 이 영역에 있는 파일들은 '스테이지됨' 상태가 되며, Git은 오직 이 스테이징된 내용만을 커밋 대상으로 간주합니다. 마지막으로, git commit 명령어를 실행하여 스테이징 영역에 준비된 변경 사항들의 스냅샷을 로컬 저장소에 영구적으로 기록합니다. 이렇게 로컬 저장소에 저장된 파일들은 '커밋됨' 상태가 되며, 각 커밋은 프로젝트의 특정 시점 상태를 나타내는 불변의 기록이 됩니다.

이러한 과정을 반복함으로써, 여러분은 프로젝트의 모든 변경 이력을 체계적으로 관리하고, 필요할 때마다 특정 시점으로 돌아갈 수 있는 강력한 능력을 갖게 됩니다.

3.6 정리 및 다음 단계

이 장에서는 Git을 사용하여 로컬 저장소를 시작하고, 파일 변경 사항을 추적하며, 의미 있는 커밋을 통해 프로젝트의 스냅샷을 기록하는 방법을 배웠습니다.

핵심 명령어 및 개념: * git init: 새로운 Git 저장소를 초기화합니다. * git status: 현재 작업 디렉토리와 스테이징 영역의 상태를 확인합니다. * git add: 작업 디렉토리의 변경 사항을 스테이징 영역으로 옮깁니다. * git commit: 스테이징 영역의 스냅샷을 로컬 저장소에 영구적으로 기록합니다. * Git의 세 가지 상태: 수정됨(Modified), 스테이지됨(Staged), 커밋됨(Committed). * Git의 세 가지 영역: 작업 디렉토리(Working Directory), 스테이징 영역(Staging Area), 로컬 저장소(Local Repository).

이제 여러분은 Git으로 프로젝트를 시작하고 기본적인 변경 이력을 기록할 수 있게 되었습니다. 하지만 이렇게 쌓인 커밋들을 어떻게 확인하고, 만약 잘못된 커밋을 했다면 어떻게 되돌릴 수 있을까요? 다음 장에서는 이 질문에 대한 답을 찾아보겠습니다. Git의 변경 이력을 조회하고, 필요에 따라 과거로 돌아가는 방법을 배우면서 Git의 시간 여행 능력을 탐험할 것입니다.
