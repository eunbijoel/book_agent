## 1. 현재 시스템 아키텍처 분석

### 1.1 전체 워크플로우

```
사용자 입력 (toc/*.json)
    │
    ▼
┌─────────────────────┐
│  Spec 검증 & 설정    │  validate_spec() → _config()
│  Grounding 해소     │  resolve_grounding()
│  목차 자동생성(선택)  │  plan_outline()
└─────────┬───────────┘
          │
          ▼
    ┌─────────────────────────────────────┐
    │  챕터 루프 (chapters 순회)            │
    │                                     │
    │  ① Planner (선택) ─ 챕터 설계        │
    │       ↓                             │
    │  ② Writer ─ 초안 작성               │
    │       ↓                             │
    │  ③ Reviewer ─ 검수                  │
    │       ↓                             │
    │  ④ 판정 게이트 ─ score < 90 ?       │
    │       │                             │
    │       ├─ NO → 저장                  │
    │       │                             │
    │       └─ YES → 수정 루프 (최대 3회)  │
    │            ⑤ Reviser ─ 수정         │
    │            ⑥ Re-Reviewer ─ 재검수    │
    │            ⑦ 개선 여부 판정           │
    │                                     │
    │  저장 → GitHub 푸시 → 요약 누적       │
    └─────────────────────────────────────┘
          │
          ▼
    PDF 생성 → README 갱신 → 완료
```

### 1.2 핵심 설계 특징


| 항목            | 현재 구현                                          |
| ------------- | ---------------------------------------------- |
| **모델**        | `gemma4:31b` 단일 모델 (모든 에이전트 공유)                |
| **컨텍스트**      | 32,768 토큰                                      |
| **온도**        | 역할별 차등 (0.2~0.8)                               |
| **구조화 출력**    | Planner/Reviewer만 Pydantic 스키마 강제              |
| **품질 게이트**    | TARGET_SCORE=90, QUALITY_GATE=80, MAX_PASSES=3 |
| **상태 관리**     | 인메모리 summaries + 파일시스템 로그                      |
| **Grounding** | file/url/text 지원, 캐시 + 수치 교차검증                 |

## 2. 에이전트별 상세 분석

### 2.1 Outline Planner (목차 생성)


| 항목        | 내용                                               |
| --------- | ------------------------------------------------ |
| **목적**    | chapters가 없을 때 책 설명/grounding으로 목차 자동 설계         |
| **입력**    | config (책 설정) + grounding_text                   |
| **출력**    | `OutlinePlan` → `[{number, title, description}]` |
| **온도**    | 0.5                                              |
| **호출 방식** | 자유 텍스트 → JSON 파싱 → Pydantic 검증 (제약 디코딩 대신)       |
| **현재 모델** | gemma4:31b                                       |
| **의존성**   | grounding.py (참고 자료 해소)                          |
| **실패 모드** | JSON 파싱 실패, 챕터 수 불일치, 논리적 흐름 부재                  |


**프롬프트 핵심**: 참고 자료가 있으면 자료의 핵심 주제/흐름 우선 반영. 앞→뒤로 누적되는 학습/논리 흐름. JSON만 출력.

### 2.2 Chapter Planner (챕터 설계)


| 항목        | 내용                                                                     |
| --------- | ---------------------------------------------------------------------- |
| **목적**    | 챕터별 본문 흐름 사전 설계 (thesis, steps, avoid)                                 |
| **입력**    | config + chapter + previous_summaries + grounding_text                 |
| **출력**    | `UnitPlan` → `{unit_id, thesis, reader_takeaway, steps[3-8], avoid[]}` |
| **온도**    | 0.3                                                                    |
| **호출 방식** | `call_structured()` (Pydantic 스키마 강제 + 자동 재시도)                         |
| **현재 모델** | gemma4:31b                                                             |
| **의존성**   | 이전 챕터 summaries                                                        |
| **실패 모드** | 스키마 미수렴 (ConvergenceError), 설계가 너무 추상적/기계적                             |


**프롬프트 핵심**: "체크리스트가 아니라 하나의 글 흐름이 보이는 설계도". 문서 유형별 흐름 차별화 (기술서→논리, 교재→학습, 소설→사건).

### 2.3 Writer (집필자)


| 항목        | 내용                                                            |
| --------- | ------------------------------------------------------------- |
| **목적**    | 완성도 높은 본문 초안 생성                                               |
| **입력**    | config + chapter + previous_summaries + grounding_text + plan |
| **출력**    | 마크다운 자유 텍스트 (## 소제목부터 시작)                                     |
| **온도**    | 0.8                                                           |
| **호출 방식** | `_call()` (자유 텍스트)                                            |
| **현재 모델** | gemma4:31b                                                    |
| **의존성**   | Planner 출력 (선택), summaries (이전 맥락)                            |
| **실패 모드** | 할루시네이션, 반복적 서술, grounding 무시, 기계적 나열식 글                       |


**프롬프트 핵심**:

- writing_guidelines 최우선
- 설계의 thesis/steps 핵심 의도 반영하되 자연스러운 흐름 우선
- 참고 자료 핵심 내용 우선 반영, 자료 없는 수치 단정 금지
- 이전 내용과의 연결 + 반복 방지

### 2.4 Review (검수자)


| 항목        | 내용                                                                                               |
| --------- | ------------------------------------------------------------------------------------------------ |
| **목적**    | 원고의 오류 검출 + 품질 다축 평가                                                                             |
| **입력**    | config + chapter + draft + grounding_text + plan                                                 |
| **출력**    | `ReviewResult` → `{needs_revision, score, quality(8축), issues[], summary, unverified_numbers[]}` |
| **온도**    | 0.2                                                                                              |
| **호출 방식** | `call_structured()` (Pydantic 스키마 강제)                                                            |
| **현재 모델** | gemma4:31b                                                                                       |
| **의존성**   | Writer 출력 (draft), grounding (수치 교차검증)                                                           |
| **실패 모드** | 스키마 미수렴, 자기 관대 편향, 할루시네이션 미검출, 점수 인플레이션                                                          |


**프롬프트 핵심**:

- 2단계 검수: ① 오류/위반(6종) → ② 품질(6종)
- 8축 품질 점수 (accuracy, completeness, clarity, depth, structure, persuasiveness, creativity, tone_fit)
- "트집이 아닌 독자 경험 최대 개선 3~5개 집중"
- 점수↔issue 일관성 강제 (85점 미만 축 → 해당 issue 필수)

### 2.5 Revise (수정)


| 항목        | 내용                                                             |
| --------- | -------------------------------------------------------------- |
| **목적**    | 검수 결과 반영하여 원고 수정                                               |
| **입력**    | config + chapter + draft + review_json + grounding_text + plan |
| **출력**    | 마크다운 자유 텍스트 (수정된 원고)                                           |
| **온도**    | 0.5                                                            |
| **호출 방식** | `_call()` (자유 텍스트)                                             |
| **현재 모델** | gemma4:31b                                                     |
| **의존성**   | Writer 출력 + Reviewer 출력                                        |
| **실패 모드** | 과도한 수정으로 원문 장점 훼손, 수정 지시 무시, 새로운 오류 도입                         |


**프롬프트 핵심**:

- 이중 전략: "고침"(위반 0까지) + "끌어올림"(품질 한 단계씩)
- 좋은 부분 최대 유지
- unverified_numbers 교체/제거
- 최종 원고만 출력 (설명 금지)

## 3. 인지 과업 분석

### 3.1 Writer 에이전트 — 필요 인지 능력


| 능력               | 중요도   | 설명                                   |
| ---------------- | ----- | ------------------------------------ |
| **한국어 유창성**      | ★★★★★ | 모든 출력이 한국어. 자연스러운 문체와 어휘 선택 필수       |
| **장문 생성 능력**     | ★★★★★ | 챕터 단위(수천 자) 일관된 품질의 긴 텍스트 생성         |
| **지식 합성**        | ★★★★☆ | grounding 자료 + 일반 지식을 통합하여 새로운 설명 구성 |
| **구조적 서술**       | ★★★★☆ | 논리적 흐름, 단계적 전개, 소제목 구성               |
| **창의성/스토리텔링**    | ★★★★☆ | 비유, 예시, 장면 묘사로 글의 흥미 유지              |
| **지시 따르기**       | ★★★★☆ | writing_guidelines, plan 설계를 정확히 반영  |
| **Grounding 준수** | ★★★☆☆ | 참고 자료 수치/사실을 정확히 인용하되 과장/왜곡 금지       |
| **맥락 연속성**       | ★★★☆☆ | 이전 챕터와 자연스럽게 연결, 반복 방지               |


**핵심 요구**: "유창한 한국어로 긴 텍스트를 자연스럽게 쓸 수 있는 **창의적 글쓰기** 능력"

### 3.2 Reviewer 에이전트 — 필요 인지 능력


| 능력            | 중요도   | 설명                             |
| ------------- | ----- | ------------------------------ |
| **오류 감지**     | ★★★★★ | 사실 오류, 논리 오류, 주제 이탈을 정확히 포착    |
| **할루시네이션 탐지** | ★★★★★ | grounding에 없는 수치/사실의 확정적 단정 식별 |
| **비판적 추론**    | ★★★★★ | 글의 약점을 구조적으로 분석하고 우선순위화        |
| **일관성 검증**    | ★★★★☆ | 앞뒤 논리, 설계-본문 일치, 자료-본문 일치 확인   |
| **누락 탐지**     | ★★★★☆ | 챕터 설명/설계에 있지만 본문에 빠진 내용 발견     |
| **다축 평가**     | ★★★★☆ | 8개 품질 축을 독립적으로 평가하고 점수 부여      |
| **구조화 출력**    | ★★★☆☆ | ReviewResult JSON 스키마를 정확히 준수  |
| **한국어 이해**    | ★★★☆☆ | 평가 대상 텍스트가 한국어이므로 정확한 이해 필요    |


**핵심 요구**: "**비판적 사고력**과 **추론 능력**. 글을 쓰는 능력보다 분석하는 능력이 중요"

### 3.3 Reviser 에이전트 — 필요 인지 능력


| 능력            | 중요도   | 설명                            |
| ------------- | ----- | ----------------------------- |
| **지시 반영 정확도** | ★★★★★ | review issues를 하나하나 정확히 반영    |
| **편집 능력**     | ★★★★★ | 기존 텍스트를 자연스럽게 수정하면서 흐름 유지     |
| **원문 보존**     | ★★★★☆ | 문제 없는 좋은 부분을 훼손하지 않고 유지       |
| **일관성 유지**    | ★★★★☆ | 수정 후에도 전체 흐름과 문체 일관성 보장       |
| **피드백 통합**    | ★★★★☆ | 여러 이슈(위반+품질)를 동시에 반영하는 복합 편집  |
| **단계적 개선**    | ★★★☆☆ | "한 번에 완벽" 대신 "한 단계씩" 끌어올리는 절제 |
| **한국어 유창성**   | ★★★★☆ | 수정된 텍스트도 자연스러운 한국어여야 함        |
| **가독성 향상**    | ★★★☆☆ | 명료성, 설득력, 깊이를 실질적으로 개선        |


**핵심 요구**: "**정밀한 지시 이행**과 **편집 능력**. 글을 새로 쓰는 것이 아니라 기존 글을 개선하는 과업"

## 4. 동일 모델 사용의 취약점 분석

현재 시스템은 **gemma4:31b 단일 모델**을 Writer, Reviewer, Reviser 모두에 사용한다. 이로 인해 다음과 같은 구조적 약점이 발생한다.

### 4.1 공유 할루시네이션 (Shared Hallucination)

**문제**: Writer가 grounding에 없는 수치를 "그럴듯하게" 생성하면, **같은 모델인 Reviewer도 해당 수치를 "그럴듯하다"고 판단**할 가능성이 높다.

```
Writer (gemma4:31b): "반도체 공정에서 EUV 노광의 정밀도는 0.33nm이다" (실제 자료에 없음)
Reviewer (gemma4:31b): accuracy 점수 88점 → unsupported_claim으로 잡지 않음
```

**원인**: 같은 학습 데이터에서 같은 사전지식을 공유하므로, 모델이 "알고 있다고 생각하는" 잘못된 정보를 Writer가 쓰고 Reviewer가 통과시킴.

**코드 수준 완화책**: `unverified_numbers()` 함수(grounding.py)가 수치를 교차검증하지만, **텍스트 형태의 사실 오류**는 여전히 사각지대.

### 4.2 공유 추론 실패 (Shared Reasoning Blind Spot)

**문제**: 특정 논리 구조에서 모델이 체계적으로 오류를 범하면, Reviewer 역시 같은 논리 결함을 가지므로 감지하지 못함.

```
Writer: "A이므로 B이고, B이므로 C이다" (실제로 A→B 연결이 논리적으로 타당하지 않음)
Reviewer: logical_error를 잡지 못하고, structure 점수 85점 부여
```

**원인**: 같은 모델은 같은 추론 편향(bias)을 공유. 자신이 쓴 논리를 자신이 검증하는 것과 유사한 맹점.

### 4.3 동일 문체/패턴 고착 (Style Lock-in)

**문제**: Writer와 Reviser가 같은 모델이므로, 수정 후에도 **동일한 문체 패턴이 반복**됨. Reviewer가 tone_problem이나 creativity_problem을 지적해도, Reviser가 본질적으로 다른 스타일을 제시하지 못함.

```
Writer: "먼저 ~를 살펴보겠습니다. 다음으로 ~를 알아보겠습니다. 마지막으로 ~"
Reviewer: structure_problem 지적 → "기계적 나열식 구성"
Reviser: "~에 대해 알아보겠습니다. 이어서 ~를 살펴봅시다. 끝으로 ~" (본질적으로 같은 패턴)
```

**결과**: 수정 루프가 3회 돌아도 tone_fit이나 creativity 점수가 올라가지 않는 **천장 현상**이 빈번.

### 4.4 자기 관대 편향 (Self-Leniency Bias)

**문제**: 같은 모델이 자신의 출력을 평가할 때, **체계적으로 높은 점수를 부여**하는 경향.

**증거**: 현재 코드에서 `TARGET_SCORE = 90`, `QUALITY_GATE = 80`을 설정한 것은 이 편향을 전제로 한 설계. 실제로 초기 검수에서 90점 이상이 자주 나온다면 이는 검수 자체의 엄격성 부족을 시사.

### 4.5 누락 컨텐츠 사각지대

**문제**: Writer가 특정 주제를 "중요하지 않다"고 판단해 건너뛰면, 같은 지식 체계를 가진 Reviewer도 같은 판단으로 `missing_content`를 잡지 못함.

### 4.6 요약: 다중 모델이 해결하는 것


| 문제        | 단일 모델    | 다중 모델                        |
| --------- | -------- | ---------------------------- |
| 공유 할루시네이션 | 감지 실패    | 다른 학습 데이터로 교차 검증             |
| 공유 추론 편향  | 같은 편향 반복 | 다른 추론 경로로 보완                 |
| 문체 고착     | 수정 무효    | Writer/Reviser가 다른 스타일 제시 가능 |
| 자기 관대 편향  | 과대 평가    | 독립적 평가로 객관성 확보               |

## 6. 에이전트-모델 매핑 권장안

### 6.1 최우선 권장 구성: 3-모델 체제

```
┌─────────────┐     ┌───────────────────┐     ┌─────────────┐
│   Writer     │     │    Reviewer        │     │   Reviser    │
│ Qwen3-32B   │ ──→ │ DeepSeek-R1:32b   │ ──→ │ Qwen3-32B   │
│ t=0.8       │     │ t=0.2             │     │ t=0.5        │
│ 자유 텍스트   │     │ 구조화 출력         │     │ 자유 텍스트   │
└─────────────┘     └───────────────────┘     └─────────────┘
```

#### Writer → **Qwen3-32B**

**선정 이유**:

1. **창의적 글쓰기 최강**: Creative Writing v3 벤치마크 84.4점으로 동급 최고
2. **한국어 최우수**: 119개 언어 지원, CJK 특화 학습 데이터, 한국어 자연스러움 검증
3. **256K 컨텍스트**: 긴 챕터 생성 + 이전 요약 + grounding 자료를 여유 있게 수용
4. **유연한 생성 모드**: thinking/non-thinking 전환으로 기술서(사고형)와 창작물(직접형) 차별화 가능
5. **Gemma4:31b 대비**: 글쓰기 벤치마크에서 우위, 특히 서사/대화 생성에서 차이 확인됨

**현재 모델(gemma4:31b) 대비 개선점**:

- 창의적 글쓰기 품질 향상 → creativity/tone_fit 점수 개선 기대
- 한국어 유창성 향상 → 자연스러운 문체로 수정 루프 감소 기대
- 문체 다양성 증가 → "기계적 나열" 패턴 감소

#### Reviewer → **DeepSeek-R1 Distilled 32B**

**선정 이유**:

1. **사고 체인 노출**: `<think>` 태그로 추론 과정이 드러나므로 **왜 문제인지** 구조적으로 분석
2. **독립적 추론 경로**: Qwen/Gemma와 완전히 다른 학습 데이터·추론 방식으로 **교차 검증** 효과
3. **강화학습 기반 추론**: R1의 RL 학습이 비판적 분석·오류 탐지에 특화
4. **할루시네이션 탐지**: Writer(Qwen)와 다른 사전지식 → **공유 할루시네이션 문제 해소**
5. **구조화 출력 호환**: JSON 스키마 출력도 양호 (Q5_K_M 이상 권장)

**현재 모델(gemma4:31b) 대비 개선점**:

- 자기 관대 편향 제거 → 더 엄격하고 객관적인 평가
- 추론 편향 교차 보완 → Writer가 놓친 논리 오류 검출률 향상
- 할루시네이션 교차 감지 → unsupported_claim 검출 강화

#### Reviser → **Qwen3-32B** (Writer와 동일 모델)

**선정 이유**:

1. **지시 따르기 최우수**: ReviewResult의 issues를 정확히 반영하는 데 필수적인 능력
2. **한국어 편집 품질**: 수정 후에도 자연스러운 한국어 유지
3. **원문 보존 능력**: 긴 텍스트 이해 + 부분 수정에 적합한 256K 컨텍스트
4. **Writer와 동일해도 괜찮은 이유**: Reviser의 입력에는 **독립 모델(DeepSeek-R1)의 ReviewResult**가 포함되므로, "다른 시각의 피드백을 같은 필력으로 반영"하는 구조가 됨

**왜 별도 모델이 불필요한가**:

- Reviser는 "창의적 글쓰기"보다 "지시 이행"이 핵심
- 핵심 교차검증은 Writer↔Reviewer 사이에서 이미 수행됨
- Reviser에 다른 모델을 쓰면 문체 불일치 위험이 오히려 증가

### 6.2 대안 구성: 2-모델 체제 (하드웨어 제약 시)

```
┌─────────────┐     ┌───────────────────┐     ┌─────────────┐
│   Writer     │     │    Reviewer        │     │   Reviser    │
│ Gemma4:31b  │ ──→ │ DeepSeek-R1:32b   │ ──→ │ Gemma4:31b  │
│ (현재 유지)   │     │ (Reviewer만 교체)   │     │ (현재 유지)   │
└─────────────┘     └───────────────────┘     └─────────────┘
```

**장점**: 현재 코드 변경 최소화. Reviewer만 MODEL 분리하면 됨.
**효과**: 가장 큰 개선점인 "교차 검증"을 최소 비용으로 달성.

### 6.3 경량 구성: MoE 활용 (VRAM 12-16GB 환경)

```
Writer:   Qwen3-30B-A3B  (3.3B 활성, ~17GB)
Reviewer: DeepSeek-R1:14b (14B, ~12GB)
Reviser:  Qwen3-30B-A3B  (Writer와 동일)
```

**장점**: 중급 GPU에서도 운용 가능. 속도 우수 (~34 tok/s).  
**주의**: 14B Reviewer의 검수 깊이가 32B 대비 약할 수 있음.
